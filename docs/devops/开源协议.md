# 开源协议

## 一、什么是开源协议

开源协议（Open Source License）是软件作者授权他人使用、修改、分发其代码的法律条款。不同协议对"能做什么、不能做什么、必须做什么"的要求各不相同。

---

## 二、核心概念

| 概念 | 说明 |
|------|------|
| **著作权（Copyright）** | 代码默认归作者所有，未经授权不得使用 |
| **Copyleft** | 要求衍生作品必须使用相同协议发布（传染性） |
| **宽松协议（Permissive）** | 允许闭源、商用，几乎无限制 |
| **专利授权** | 部分协议明确授予专利使用权，防止专利诉讼 |

---

## 三、常见开源协议一览

### 1. MIT

> 最宽松、最流行的协议之一。

- **允许**：商业使用、修改、分发、私有化
- **要求**：保留原始版权声明和许可证文本
- **禁止**：无
- **代表项目**：React、Vue.js、jQuery、Rails

```
Permission is hereby granted, free of charge, to any person obtaining a copy...
```

---

### 2. Apache 2.0

> 比 MIT 多了专利保护条款。

- **允许**：商业使用、修改、分发、专利使用
- **要求**：保留版权声明、说明修改内容（NOTICE 文件）
- **禁止**：使用作者商标
- **亮点**：明确授予专利许可；若对方发起专利诉讼，则自动终止其授权
- **代表项目**：Kubernetes、Hadoop、Kafka、Android（部分）

---

### 3. GPL v2 / GPL v3

> 强 Copyleft，具有"传染性"。

- **允许**：商业使用、修改、分发
- **要求**：衍生作品必须以 GPL 开源；分发二进制时须提供源码
- **禁止**：闭源分发
- **v2 vs v3**：v3 增加了反 Tivoization（禁止硬件锁定）和专利报复条款
- **代表项目**：Linux Kernel（v2）、Git（v2）、WordPress（v2）、GCC（v3）

::: warning 注意
使用 GPL 代码的项目，整体必须以 GPL 发布，**不能商业闭源**。
:::

---

### 4. LGPL（Lesser GPL）

> GPL 的弱化版，专为库设计。

- **允许**：商业软件**动态链接** LGPL 库而无需开源
- **要求**：对 LGPL 库本身的修改必须开源
- **适用场景**：希望库被商业项目使用，但仍保持库本身开源
- **代表项目**：Qt（LGPL 版）、FFmpeg（部分）

---

### 5. AGPL v3

> GPL 的加强版，堵住"网络漏洞"。

- **要求**：通过网络提供服务（SaaS）也必须开放源码
- **适用场景**：防止云厂商"白嫖"开源代码而不回馈
- **代表项目**：MongoDB（早期）、Grafana、Nextcloud

---

### 6. BSD（2-Clause / 3-Clause）

> 与 MIT 类似，极为宽松。

- **2-Clause（Simplified BSD）**：保留版权声明即可，等同 MIT
- **3-Clause（New BSD）**：在 2-Clause 基础上，**禁止**用作者名字做广告宣传
- **代表项目**：FreeBSD、OpenBSD、Go 语言

---

### 7. MPL 2.0（Mozilla Public License）

> 文件级 Copyleft，折中方案。

- **要求**：修改过的 MPL 文件必须开源，但可与闭源代码混合分发
- **亮点**：Copyleft 只作用于单个文件，不传染整个项目
- **代表项目**：Firefox、Thunderbird

---

### 8. ISC

> 功能上等同 MIT/2-Clause BSD，措辞更简洁。

- **代表项目**：OpenBSD、部分 npm 包

---

### 9. CC 系列（Creative Commons）

> 主要用于**文档、图片、内容**，不推荐用于代码。

| 协议 | 说明 |
|------|------|
| CC0 | 放弃所有权利，进入公有领域 |
| CC BY | 署名即可自由使用 |
| CC BY-SA | 署名 + 相同方式共享（类似 Copyleft） |
| CC BY-NC | 署名 + 非商业使用 |
| CC BY-NC-SA | 署名 + 非商业 + 相同方式共享 |

---

### 10. 其他值得关注的协议

| 协议 | 特点 |
|------|------|
| **Unlicense** | 彻底放入公有领域，比 CC0 更简单 |
| **WTFPL** | "Do What The F*** You Want" — 极度宽松 |
| **SSPL** | MongoDB 自创，比 AGPL 更严格（云服务商须开源整个服务栈） |
| **Commons Clause** | 附加条款，禁止"销售"软件，常叠加在 Apache/MIT 上 |
| **BSL（Business Source License）** | 限时开源：若干年后自动转为开源协议（如 MariaDB） |

---

## 四、协议对比总览

| 协议 | 商业使用 | 修改 | 分发 | 专利授权 | 闭源可以 | 传染性 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|
| MIT | ✅ | ✅ | ✅ | ❌ | ✅ | 无 |
| Apache 2.0 | ✅ | ✅ | ✅ | ✅ | ✅ | 无 |
| BSD 2/3 | ✅ | ✅ | ✅ | ❌ | ✅ | 无 |
| GPL v2/v3 | ✅ | ✅ | ✅ | ✅(v3) | ❌ | 强（项目级） |
| LGPL | ✅ | ✅ | ✅ | ✅ | ✅（动态链接） | 弱（库级） |
| AGPL | ✅ | ✅ | ✅ | ✅ | ❌ | 强（含网络） |
| MPL 2.0 | ✅ | ✅ | ✅ | ✅ | ✅ | 弱（文件级） |

---

## 五、如何选择

### 决策流程

```
你想开源代码吗？
│
├─ 否 → 不用选，默认版权保留
│
└─ 是
   │
   ├─ 希望尽量宽松，别人怎么用都行？
   │   ├─ 需要专利保护 → Apache 2.0
   │   └─ 不需要         → MIT 或 BSD
   │
   ├─ 希望衍生作品也必须开源？
   │   ├─ 包括 SaaS/云服务 → AGPL v3
   │   ├─ 只是库，允许商业闭源链接 → LGPL
   │   └─ 一般项目 → GPL v2 或 GPL v3
   │
   ├─ 折中：文件级 Copyleft → MPL 2.0
   │
   └─ 文档/内容/数据集 → CC BY 或 CC BY-SA
```

---

### 场景推荐

| 场景 | 推荐协议 |
|------|----------|
| 个人项目，希望被广泛使用 | **MIT** |
| 企业开源，需要专利保护 | **Apache 2.0** |
| 基础设施/工具库，不想被云厂商闭源 | **AGPL v3** |
| 开源库，允许商业软件调用 | **LGPL** 或 **MIT** |
| 操作系统/编译器等强开放生态 | **GPL v2** |
| 既想开源又保留商业竞争力 | **BSL** 或 **Commons Clause** |
| 文档/博客内容 | **CC BY 4.0** |
| 彻底放弃版权 | **CC0** 或 **Unlicense** |

---

### 兼容性注意事项

- **GPL v2 与 GPL v3 不兼容**：不能把两者的代码混合进同一程序
- **Apache 2.0 与 GPL v2 不兼容**：因为 GPL v2 不允许额外限制（专利条款被视为额外限制）
- **Apache 2.0 与 GPL v3 兼容**
- **MIT / BSD 与 GPL 兼容**：可以将 MIT 代码并入 GPL 项目

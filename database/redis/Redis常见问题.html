Redis 常见问题

1. 如何保证Redis和数据库的数据一致性？

答：可以通过以下几种方式来保证Redis和数据库的数据一致性：

  • 使用分布式事务：在进行数据库操作时，使用事务可以确保操作要么全部成功，要么全部失败。可以在事务中先更新数据库，然后再更新Redis缓存。
  • 监听数据库事件：可以通过监听数据库的事件（如插入、更新、删除等）来触发相应的操作。当数据库数据发生变化时，监听事件可以及时更新Redis缓存，保持数据的一致性。
  • 采用Cache Aside模式：在读取数据时，先从Redis缓存中读取数据。如果缓存中不存在，则从数据库中读取数据，并将数据写入缓存。在更新数据时，先更新数据库，然后再删除Redis缓存中的数据，确保缓存和数据库的数据保持一致。
    • 如果缓存删除失败，也会数据不一致
  • 采用延迟双删+消息队列：写数据库操作之前，先删一遍缓存；写完之后，发送一条“延迟删除缓存”的消息到MQ。
    • 该方案针对的是先删缓存再更新数据库的并发问题：
      • 写请求A：先删除缓存，再更新数据库
      • 读请求B：在写请求更新数据库之前读取数据（缓存已删），从 DB 加载旧数据，并回写缓存。此时写请求才更新数据库，这样导致DB和缓存数据不一致。

建议使用：

  • CDC（更新数据捕获工具）：Canal，监听数据库更新事件工具，并将事件发送给消息队列
  • 消息队列：Kafka

2. RDB和AOF持久化的区别？

  • RDB：保存的是某一刻的内存快照，保存到硬盘
  • AOF：保存的是执行命令，当数据保存后，将收到的命令记录到日志。一般与RDB配合使用。

3. Redis集群架构

模式主从结构数据分布水平扩展故障检测
主从模式1主1从全量存储，无分片
集群模式3主3从（主节点>=3）分片存储（16384 个槽位）支持，增加主节点即可扩展容量和吞吐主节点间通过 Gossip 协议 + 投票
哨兵模式1主多从+3个哨兵（建议奇数）全量存储，无分片不支持，只能扩展从节点读能力哨兵节点监控 + 投票选举新主节点
  • 单主机
  • 1主1从 = 数据冗余 + 读写分离
  • 1主1从3哨兵 = 数据冗余 + 读写分离 + 高可用（自动故障转移）
  • 3主3从 = 数据冗余 + 读写分离 + 高可用（自动故障转移） + 数据分片

4. 缓存穿透 、 缓存穿击 、 缓存雪崩 、 缓存污染

（1）缓存穿透：
问题来源：用户查缓存和数据库均不存在的数据，缓存不命中，导致请求直接落到数据库，流量大时，DB可能就会挂掉。

解决方法：

  1. 接口层做校验，对于不存在的查询条件直接返回空
  2. 缓存空对象
  3. 布隆过滤器

（2）缓存击穿：
问题来源：用户查缓存没有、数据库有的数据，缓存不命中，导致请求直接落到数据库，流量大时，DB可能就会挂掉。

解决方法：

  1. 热点数据永不过期+定时任务更新
  2. 逻辑过期+异步更新
  3. 接口限流与熔断，降级
  4. 对“查数据库”那段逻辑加互斥锁

（3）缓存雪崩：
问题来源：大量缓存同一时间过期，导致请求直接落到数据库

解决方法：

  1. 热点数据永不过期+定时任务更新
  2. 随机设置过期时间

5. 什么是布隆过滤器？

布隆过滤器是一种空间效率极高的概率性数据结构，用于判断一个元素是否存在于集合中。

使用方法：

  1. 先为所有key生成布隆过滤器
  2. 每次取缓存时，先到布隆过滤器检查是否有该key，没有则直接返回，省去查缓存和数据库

6. 键过期的删除策略

  • 惰性删除：服务器不会主动删除数据，只有当客户端查询某个数据时，服务器判断该数据是否过期，如果过期则删除。
  • 定期删除：服务器执行定时任务删除过期数据，但是考虑到内存和CPU的折中（删除会释放内存，但是频繁的删除操作对CPU不友好）， 该删除的频率和执行时间都受到了限制。

7. 内存淘汰策略有哪些

  • 不淘汰 noeviction （v4.0后默认的）
  • 对设置了过期时间的数据中进行淘汰
    • 随机： volatile-random
    • ttl： volatile-ttl
    • lru： volatile-lru
    • lfu： volatile-lfu
  • 全部数据进行淘汰
    • 随机： allkeys-random
    • lru： allkeys-lru
    • lfu： allkeys-lfu

LRU算法：全称是 Least Recently Used。Redis为每个键（key）记录一个 LRU 时间戳（最后一次被访问的时间，精确到毫秒） 当需要淘汰键时，Redis 会随机采样 N 个键（默认 maxmemory-samples 5），从这些采样中选出 LRU 时间戳最旧 的键进行淘汰。

LFU算法：全称是 Least Frequently UsedLFU。在 LRU 策略基础上，为每个数据增加了一个计数器，来统计这个数据的访问次数。 LFU会先把访问次数最低的数据淘汰出缓存，若访问次数相同，则按照LRU来淘汰。

8. 哨兵模式中，哨兵的作用

  • 监控（Monitoring）：哨兵会不断地检查主节点和从节点是否运作正常。
  • 自动故障转移（Automatic failover）：当主节点不能正常工作时，哨兵会开始自动故障转移操作，它会将失效主节点的其中一个从节点升级为新的主节点，并让其他从节点改为复制新的主节点。
  • 配置提供者（Configuration provider）：客户端在初始化时，通过连接哨兵来获得当前Redis服务的主节点地址。
  • 通知（Notification）：哨兵可以将故障转移的结果发送给客户端。
